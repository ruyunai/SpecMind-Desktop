## BUG-001: langgraph 0.2.34 缺少 interrupt() 和 Command
- **现象**：`ImportError: cannot import name 'interrupt' from 'langgraph.types'`
- **根因**：`interrupt()` 和 `Command` 在 langgraph 0.3.x 才引入，0.2.34 不可用
- **修复方案**：放弃原生 interrupt()，改用条件路由 + resume 图 + threading.Event 模式
  - 主图 Legal 高风险 → 路由到 END（阻断）
  - Orchestrator 发 workflow_blocked 信号，MainWindow 弹确认框
  - 用户确认 → Orchestrator 运行独立 resume 图（PM → fan-out → Planner）
  - 用户拒绝 → workflow_complete 终止
- **是否成功**：✅ 成功
- **涉及文件**：src/graph/builder.py, src/core/orchestrator.py, src/gui/main_window.py
- **日期**：2026-07-13

## BUG-002: 并行节点 InvalidUpdateError "Can receive only one value per step"
- **现象**：Commercial/Contract/Review 并行执行时报 `InvalidUpdateError: At key 'raw_input': Can receive only one value per step`
- **根因**：并行节点返回完整 state（含 raw_input 等相同字段），LangGraph 不允许同一 super-step 多个节点写同一字段
- **修复方案**：
  1. 重构 mock_agents.py：所有节点只返回本节点更新的字段（部分 dict），符合 LangGraph 规范
  2. 并行节点（Commercial/Contract/Review）不返回 `current_node`（会冲突）
  3. state.py 的 `audit_snapshots` 改用 `Annotated[List[Dict], operator.add]` reducer，支持并行追加合并
- **是否成功**：✅ 成功
- **涉及文件**：src/agents/mock_agents.py, src/agents/state.py
- **日期**：2026-07-13

## BUG-003: audit_snapshots 计数只有 1（应为 7）
- **现象**：工作流完成后日志显示「审计快照数: 1」，实际执行了 7 个节点
- **根因**：Orchestrator 用 `graph.get_state(config)` 从 checkpointer 拉取最终 state 覆盖本地 state，但 checkpointer 在并行场景下 audit_snapshots 可能不完整（reducer 行为不符预期）
- **修复方案**：在 `_run_main_graph` 的 `get_state` 后，对比手动累加的 audit_snapshots 数量与 checkpointer 的数量，取较大值
  ```python
  manual_snapshots = self._state.get("audit_snapshots", [])
  self._state = dict(final_state.values)
  if len(manual_snapshots) > len(self._state.get("audit_snapshots", [])):
      self._state["audit_snapshots"] = manual_snapshots
  ```
- **是否成功**：✅ 成功（验证脚本确认 7 个节点快照全部记录）
- **涉及文件**：src/core/orchestrator.py
- **日期**：2026-07-13
- **备注**：升级 langgraph-checkpoint-sqlite 到 3.1.0 后，checkpointer 本身的 audit_snapshots 也变为 7，但修复逻辑保留作为向后兼容保护

## BUG-004: ChromaDB metadata 不接受嵌套 dict
- **现象**：seed_data.py 加载报错 `ValueError: Expected metadata value to be a str, int, float, bool, SparseVector, list, or None, got {...} which is a dict`
- **根因**：`make_meta()` 返回的 AssetMeta 含 `extra` 字段（嵌套 dict，如 `{"article_no": "第十三条", "keywords": [...]}`），ChromaDB metadata 仅支持扁平结构
- **修复方案**：在 `ChromaStore.add_documents()` 写入前调用 `_flatten_meta()` 平铺元数据
  - `extra` 字段提到顶层
  - 复杂类型（嵌套 dict）序列化为 JSON 字符串
  - list 元素为基本类型时保留原样
- **是否成功**：✅ 成功（19 条种子数据全部入库）
- **涉及文件**：src/storage/chroma_store.py
- **日期**：2026-07-13

## BUG-006: doc_hash 丢失导致 RRF 合并所有结果为 1 条
- **现象**：向量召回 7 条，但 RRF 融合后只剩 1 条，Recall@5=0.06
- **根因**：`make_meta()` 默认返回 `doc_hash=""`（空字符串），`add_documents()` 中 `if "doc_hash" not in meta` 判断永远为 False（键存在但值为空），导致所有文档的 doc_hash 都是 ""，RRF 把它们当作同一条文档合并
- **修复方案**：把判断改为 `if not meta.get("doc_hash")`（空字符串也触发覆盖）
- **是否成功**：✅ 成功（RRF 后保留 5 条独立结果，Recall@5=0.94）
- **涉及文件**：src/storage/chroma_store.py
- **日期**：2026-07-13

## BUG-007: FTS5 unicode61 tokenizer 对中文短语匹配失效（BM25 全 0 召回）
- **现象**：20 条评估用例 BM25 召回全部 0 条，MATCH '人脸识别' / MATCH '数据出境' / MATCH '终身免费' 均无命中
- **根因**：`tokenize='unicode61'` 把中文按单字切分，「在线教育」会被切成「在/线/教/育」4 个单字 token，这些单字在文档中普遍存在（"的/在/一"等高频字），BM25 几乎无法区分相关性
- **修复方案**：迁移到 `tokenize='trigram'`（3 字符滑窗切分）
  1. `_init_tables` 改用 trigram tokenizer + 自动迁移逻辑（检测旧表 DROP + CREATE + 回填）
  2. `add_asset` 写入 FTS5 时把 metadata.keywords 拼接到 content 后（原 keywords 字段未进索引）
  3. `search_fts` 按 query 长度分流：
     - <3 字符：trigram 不支持，走 LIKE 模糊匹配
     - 3-5 字符：trigram MATCH（切成 1-3 个 3-gram AND，命中率高）
     - >=6 字符：按 4 字符滑窗切成多个子串，每段 trigram MATCH，结果 OR 合并（避免长 token 切成过多 3-gram AND 全失败）
- **是否成功**：✅ 成功
  - 单关键词召回：8/9 命中（仅 "BI" 2 字符走 LIKE 回退）
  - 评估 BM25 召回率：14/20 (70%)（从 0% 提升）
  - **Recall@5: 0.94 -> 1.00**（混合检索协同效应提升）
  - Prompt 安全性 1.00 / 置信度阻断 1.00 保持
- **失败后替代方案**：未尝试 jieba 预分词，trigram + 滑窗已达成 100% Recall@5
- **涉及文件**：src/storage/sqlite_store.py
- **日期**：2026-07-13
- **备注**：未采用 jieba 方案，理由：(1) 60MB 词典依赖违背本地优先原则 (2) PyInstaller bundle 复杂 (3) trigram 已达成 100% Recall@5，无升级必要

## BUG-013: llm_client.py 无 User 分隔符时 HumanMessage 为空（高危）
- **现象**：当 prompt 以 "System:" 开头但不含 "\n\nUser:" 分隔符时，三元表达式 `(prompt, "")` 返回 2 元素元组，`len(parts) > 1` 恒为 True，导致 system = 整个 prompt，user = "" 空字符串，LLM 收到空用户消息
- **根因**：`parts = prompt.split(...) if ... else (prompt, "")` 的 else 分支设计错误，2 元素元组使后续 `len(parts) > 1` 判断失效；第 105-106 行的 else 是死代码永远不会执行
- **修复方案**：改为 `if prompt.startswith("System:") and "\n\nUser:" in prompt:` 同时满足两条件才拆分；用 `system_part, user_part = prompt.split("\n\nUser:", 1)` 解构赋值，避免 len 判断
- **是否成功**：✅ 成功
- **涉及文件**：src/agents/llm_client.py
- **日期**：2026-07-14
- **备注**：当前 build_*_prompt 均不以 "System:" 开头，此分支未被触发，属隐患修复
