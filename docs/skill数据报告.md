# SpecMind Desktop 项目 Skill 使用数据报告

> 📅 报告日期：2026-07-15
> 🏷️ 项目版本：v0.2.1
> 📦 产物：SpecMindDesktop.exe（130.52 MB 单文件）
> 🤖 模型：GLM-5.1

## 一、报告概述

本报告系统总结 SpecMind Desktop 项目开发过程中实际使用的 4 个 AI Skill 的功能定位、辅助成效、量化数据与优化空间。SpecMind Desktop 是一款 ToB 企业级本地 Windows 桌面应用，使用 7 个 LangGraph Agent 协同将脏销售需求在 15 分钟内转化为标准化 PRD + 报价 + 合同比对 + 评审 + 交付计划，技术栈为 PySide6 + LangGraph + ChromaDB + DeepSeek API + PyInstaller。

本次开发周期内累计修复 **32 个 Bug**（BUG-001 ~ BUG-032，含 6 个致命级），完成 **21 个测试套件**，RAG 召回率从 0.76 提升至 0.92，Prompt 工程化质量从 0.60 提升至 1.00。4 个 Skill 在记忆管理、科学调试、RAG 审查、LangGraph 开发四个维度为项目提供了关键支撑。

## 二、4 个 Skill 详细说明

### 2.1 🧠 记忆维护（memory_maintenance）

| 维度 | 说明 |
|------|------|
| **功能定位** | 项目记忆库的维护技能，负责 memory-bank 下 7 个文件的初始化、日常更新与 bug 修复追加 |
| **核心能力** | 7 文件初始化（项目概要/产品背景/架构设计/技术环境/当前状态/进度追踪/修复日志）、状态/进度同步、修复日志按 BUG-XXX 编号递增、IDE 规则适配 |
| **适用场景** | 项目启动初始化、会话切换上下文恢复、Bug 修复归档、里程碑节点更新 |
| **触发时机** | 项目主流程阶段 0 自动调用初始化，后续日常维护独立触发 |

**在本次项目中担任的角色**：

记忆维护是整个项目的"知识中枢"，从阶段 0 初始化 7 个记忆文件开始，全程伴随 12 个里程碑（M0 ~ M10）的推进。32 个 Bug 从 BUG-001（langgraph 0.2.34 缺少 interrupt）到 BUG-032（exe 构建环境错误）全部按规范记录，每条包含：现象 / 根因 / 修复方案 / 是否成功 / 涉及文件 / 日期 / 备注。会话切换时通过阅读修复日志避免重复尝试已失败方案（如 jieba 分词方案被明确标记为不采用）。

**具体辅助完成的开发任务**：
- 初始化 7 个记忆文件 + IDE 规则文件（.trae/rules/project_rules.md）
- 维护 12 个里程碑状态（M0 ~ M10，其中 M8/M9/M11/M12 待开始）
- 记录 32 个 Bug 完整生命周期（6 致命 + 17 高危 + 9 中低危）
- 同步 21 个测试套件的通过状态
- 归档 3 次构建失败原因与修复方案

### 2.2 🔬 TRAE-debugger（科学调试）

| 维度 | 说明 |
|------|------|
| **功能定位** | 科学调试流程技能，通过 HTTP 收集运行时日志，遵循「假设 → 插桩 → 复现 → 分析 → 修复 → 验证」六步法 |
| **核心能力** | 启动调试服务器收集运行时证据、假设驱动的调试流程、静态分析无法定位的复杂问题诊断 |
| **适用场景** | 运行时 Bug 诊断、环境相关崩溃、打包后行为异常、多层级连锁失败 |
| **触发时机** | 用户主动要求运行时调试、或经多轮静态分析仍无法解决问题时 |

**在本次项目中担任的角色**：

TRAE-debugger 在项目后期打包阶段发挥了关键作用，专门处理"源码正常但 exe 异常"的环境相关致命问题。这类问题的共同特点是：静态代码审查完全看不出问题，必须通过运行时证据才能定位根因。

**具体辅助完成的开发任务**：
- **BUG-030（致命）**：exe 打包后 LLM 调用失败静默回退 mock 数据。通过运行时日志收集发现 3 层连锁失败：① SSL 证书缺失（spec 未打包 certifi cacert.pem）② API Key 路径不一致（源码模式 vs %APPDATA% 路径）③ 静默回退 mock（`except Exception: pass`）。修复后 exe 正确调用 LLM 并显式标注失败
- **BUG-032（致命）**：exe 构建环境错误，系统 Python 无 PySide6 导致 ModuleNotFoundError。通过构建日志分析定位到 Python 环境混淆（.venv vs 系统 Python），修复为固定使用 `.venv\Scripts\python.exe`
- **BUG-019（高危）**：elapsed_ms 测量恒为近零。通过运行时插桩确认 LangGraph 0.2.x updates 模式的信号滞后特性，重构为节点内部测量耗时

### 2.3 📊 RAG审查（RAG_review）

| 维度 | 说明 |
|------|------|
| **功能定位** | RAG 知识库 12 维度查漏补缺审查，输出问题清单 + 优化方案，用户确认后执行修复 |
| **核心能力** | 12 维度全覆盖审查（数据治理/分块策略/解析层/向量库/关系库/混合检索/查询改写/Prompt 安全/置信度/评估体系/RAG Agents/种子数据）、问题分级（致命/高危/中等/低优）、优化方案生成 |
| **适用场景** | RAG 系统质量提升、召回率优化、Prompt 工程化、检索置信度治理 |
| **触发时机** | 项目主流程阶段 8 调用，或独立触发用于 RAG 质量审查 |

**在本次项目中担任的角色**：

RAG 审查是项目 RAG 子系统质量跃升的核心驱动力。在阶段 8.RAG 中执行了完整的 12 维度审查，发现 11 个问题（致命 2 / 高危 3 / 中等 3 / 低优 3），全部修复后三指标达标。

**具体辅助完成的开发任务**：
- 12 维度审查报告生成（11 个问题分级清单）
- 数据治理：schema.py 元数据规范 + is_expired 2 年过期检查
- 分块策略：chunker.py 结构化分块（法条按条/合同按款/PRD 按 8 模块）
- 混合检索：retriever.py 向量 + BM25 + RRF（k=60）融合 + 阈值 0.6
- FTS5 分词迁移：unicode61 → trigram（3 字符滑窗），BM25 召回率 0% → 70%
- 查询改写：query_rewriter.py 三 Agent 关键词字典扩展 + 方案 D LLM 增强查询改写
- Prompt 安全：防注入 + 引用溯源 + 低置信度兜底 + KB 优先于 LLM 优先级修正
- 评估体系：eval_dataset.json 25 条 + 三指标评测（Recall@5 / Prompt / 置信度）

### 2.4 🔄 langgraph开发（langgraph_dev）

| 维度 | 说明 |
|------|------|
| **功能定位** | LangGraph Agent 专项开发，强制先拆节点 + 画 Mermaid 流程图让用户审核，再逐项选型 |
| **核心能力** | 节点拆分与拓扑设计、Mermaid 流程图可视化审核、逐项选型（LLM/RAG/记忆/工具/可观测性/人工介入）、State 结构设计（含 Annotated reducer）、条件路由与 Interrupt 实现 |
| **适用场景** | 多 Agent 工作流设计、StateGraph 构建、并行节点编排、人工介入阻断机制 |
| **触发时机** | 项目主流程阶段 8 调用，用于 LangGraph 专项开发 |

**在本次项目中担任的角色**：

langgraph 开发是 7 节点工作流的架构基石。通过强制"先拆节点 + 画图审核 + 逐项选型"流程，杜绝了黑盒直奔编译的风险，确保工作流设计在编码前经过用户确认。

**具体辅助完成的开发任务**：
- 7 节点工作流设计：SAR → Legal → [Interrupt] → PM → fan-out（Commercial/Contract/Review）→ Planner
- Mermaid 流程图审核（用户确认拓扑结构后再编码）
- 逐项选型：LLM 路由（7 Agent 按角色路由 DeepSeek）、RAG 增强（SAR/Legal/Contract）、记忆（SqliteSaver）、可观测性（audit_snapshots）、Interrupt（Legal 高风险阻断）
- State 结构设计：SpecMindState TypedDict + Annotated[List, operator.add] reducer（并行安全）
- 条件路由实现：Legal 高风险 → END（阻断）/ 低风险 → PM（继续）
- Interrupt 机制：langgraph 0.2.34 无原生 interrupt()，改用条件路由 + resume 图 + threading.Event 模式

## 五、可优化空间分析

### 5.1 🧠 记忆维护 — 优化空间

| 不足之处 | 改进方向 |
|---------|---------|
| 7 文件手动同步易遗漏 | 增加自动一致性检查：阶段更新时自动校验「当前状态」与「进度追踪」的里程碑状态是否一致 |
| 修复日志累积后查阅成本高 | 增加按严重等级/涉及文件/Skill 分类索引，支持快速检索历史方案 |
| 缺少 Bug 关联分析 | 增加「相似 Bug 推荐」功能：新 Bug 提交时自动匹配历史相似案例 |
| 无回滚记忆快照 | 增加记忆文件版本快照，支持回退到任意历史状态 |

### 5.2 🔬 TRAE-debugger — 优化空间

| 不足之处 | 改进方向 |
|---------|---------|
| 调试服务器需手动启动 | 增加自动插桩模式：检测到多轮静态分析无果时主动建议启动调试器 |
| 仅支持 HTTP 日志收集 | 扩展支持 PyInstaller 打包后 exe 的运行时调试（注入日志钩子） |
| 假设需人工提出 | 增加 LLM 辅助假设生成：基于错误现象自动生成 Top-3 假设清单 |
| 缺少环境差异检测 | 增加环境对比功能：自动对比源码模式与打包模式的路径/依赖/证书差异 |
| 验证步骤需手动执行 | 增加自动回归验证：修复后自动运行相关测试套件确认无回归 |

### 5.3 📊 RAG审查 — 优化空间

| 不足之处 | 改进方向 |
|---------|---------|
| 12 维度审查一次性执行较重 | 支持增量审查：仅审查本次变更涉及的维度 |
| 优化方案需用户逐条确认 | 增加「批量自动修复」模式：对低优问题自动执行修复并生成 diff |
| 评估数据集需手动维护 | 增加评估用例自动生成：基于实际查询日志自动扩充测试集 |
| legal 分类召回率偏低（77.78%） | 增加法规同义词词典扩展 + 法条编号检索增强 |
| 缺少 Prompt 版本管理 | 增加 Prompt A/B 测试能力：对比不同 Prompt 版本的输出质量 |

### 5.4 🔄 langgraph开发 — 优化空间

| 不足之处 | 改进方向 |
|---------|---------|
| Mermaid 流程图与代码易脱节 | 增加图代码一致性校验：builder.py 节点/边变更时自动校验流程图 |
| 选型决策无成本估算 | 增加 LLM 调用成本预估：按选型方案自动计算单次工作流 Token 成本 |
| Interrupt 机制为模拟实现 | 跟踪 langgraph 0.3.x 原生 interrupt() 兼容性，适时迁移 |
| 缺少节点性能基准 | 增加节点耗时基准库：自动对比单次执行耗时与历史均值，异常时告警 |
| 并行节点调试困难 | 增加并行节点独立日志通道：每个并行节点独立日志流，便于隔离分析 |
